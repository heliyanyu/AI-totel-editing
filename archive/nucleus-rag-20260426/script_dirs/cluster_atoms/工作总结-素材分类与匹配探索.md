# 素材分类与匹配探索 · 工作总结

> 2026-04-07 ~ 04-08

## 目标

把 Nucleus 医学动画素材库（149 个视频，3347 个语句段）自动匹配到蓝图中医生口播的对应位置，生成带素材轨的剪映草稿。

---

## 一、数据概况

### 原文端（蓝图）
- 146 个蓝图，4424 个 logic blocks
- 每个 block 有两层文本：
  - `items`：LLM 概括的要点（如"情绪激动致斑块包膜破裂"）
  - `atoms`：医生实际口播原文（如"平时这些斑块只是让管道变窄，但在情绪激动极度劳累或气温骤降时，斑块表面的包膜可能会突然破裂"）
- **关键发现：items 跟 atoms 经常错位，必须用 atoms 原文做匹配**

### 素材端（Nucleus cardiology）
- 149 个视频，72GB，总时长约 8 小时
- 107 个有英文旁白，42 个纯动画无旁白
- 素材特点：旁白描述什么，画面就展示什么，一致性好

---

## 二、探索过程

### 第一轮：用 items 做分类聚类

1. 提取 4424 个 block 的 items 文本
2. Claude 分批判断是否需要医学动画 → 822 个需要，773 个唯一标签
3. 标签合并去重 → 125 个类别
4. 素材按 ASR 原始分段（3347 段）归入类别
5. 匹配：logic block → 类别 → 类别下选 segment

**结果：匹配不准。** 因为 items 是 LLM 概括，跟医生原话错位。

### 第二轮：改用 atoms 原文

1. 重新提取 atoms 原文
2. 用 atoms 重跑分类 → 774 个需要动画，761 个标签 → 152 个类别
3. 匹配时用医生中文原话对素材英文旁白

**结果：明显改善。** 中文原文对英文原文，Claude 跨语言理解能力足够。

### 第三轮：素材端语义切割

**问题：** ASR 原始分段每段 2-8 秒太碎，丢失上下文。

- **v1 语义段落切割**：按"完整医学主题"切成 710 个段落 → 太粗，一个段落包含多个动作
- **v2 Logic block 切割**：借鉴 step1.ts 的 `||` 标记法，只插边界不改原文 → 切成 1128 个 blocks，方向对但颗粒度不均匀

### 第四轮：匹配方式迭代

| 版本 | 方式 | 效果 |
|------|------|------|
| v1 | items → 类别 → 单个 segment | 不准，items 错位 |
| v2 | atoms → 类别 → Claude 自选时间范围 | 好很多，但范围选得不总是对 |
| v3 | atoms → 类别 → 语义段落 + 英文旁白匹配 | 用中文摘要匹配时效果差，改用英文原文后改善 |
| v3 speed | 同上 + 素材完整过程加速播放 | **目前最佳版本** |
| v4 | 连续 blocks 合并成组匹配 | 夹杂不相关内容，不如逐 block 匹配 |

### 第五轮：画面描述

尝试让 Claude 为每个素材 block 生成"画面上正在展示什么动作"的中文描述。方向对，但描述太粗，且本质上又回到了"中文概括对中文概括"的老路。

---

## 三、关键结论

### 什么是对的
1. **用 atoms 原文**，不用 items 概括
2. **中文原文对英文原文匹配**，不经过翻译/概括中间层
3. **`||` 标记切割法**：零信息损耗，只插边界不改原文
4. **speed 加速**：素材完整过程加速到医生讲话时长，视觉信息完整
5. **逐 block 匹配**优于组匹配
6. **分类做初筛**缩小候选范围有效

### 什么是错的
1. **标签从数据中涌现**：761 个标签会随数据量线性增长，不可持续
2. **中文摘要做匹配**：任何形式的概括都会丢信息
3. **语义切割太粗**：一个段落包含多个动作时匹配必然不精准
4. **"宁多勿少"的判断门槛**：匹配了太多不相关的素材，效果反而变差

### 核心瓶颈
1. **素材端 logic block 切割颗粒度不均匀** — 需要调 prompt 让每个 block 只对应一个视觉动作
2. **分类体系不应从数据涌现** — 应预先定义固定的分类体系，两边共用

## 四、下一步方向

### 建立固定分类体系
- 基于《文案场景语义素材清单.xlsx》的子场景扩展完善
- 作为素材端和原文端共用的标准索引
- 不会随数据量增长而膨胀

### 素材端处理
1. Logic block 切割：调 prompt 确保每个 block 只对应一个视觉动作
2. 每个 block 归入固定分类
3. 保留英文原文用于匹配

### 匹配流程
1. 原文 block → 归入分类（初筛）
2. 同一分类下，医生中文原话 vs 素材英文旁白（精选）
3. 选中的素材完整过程加速播放到对应时长
4. 生成剪映草稿到剪辑师电脑

### 工程优化
- 素材库放剪辑服务器本地，通过局域网共享给剪辑师 → 解决 P 盘读取慢的问题
- pyJianYingDraft 读取本地素材毫秒级，当前 P 盘每个 clip 要 30 秒
